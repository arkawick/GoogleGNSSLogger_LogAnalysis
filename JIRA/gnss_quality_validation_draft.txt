DESCRIPTION
================================================================================

BACKGROUND
----------
High-quality GNSS performance depends on three critical components: the GNSS
chip, oscillator, and antenna. Google provides a standardized toolset to
evaluate GNSS performance on Android devices:

  - GnssLogger App     : Collects raw GNSS measurements from the device.
  - gnss_analysis.ipynb: Google Colab notebook (v2026.01.29) that analyzes
                         collected logs against 32 conformance checks with
                         defined quality thresholds.

This ticket documents the results of GNSS quality validation performed on two
Sony devices using the above tools, in alignment with the Google Android
Bootcamp GNSS Quality Framework.


TEST SETUP
----------
  Tool (Logger)  : GnssLogger App (Android)
  Tool (Analysis): gnss_analysis.ipynb — Google Colab (v2026.01.29)
  Test type      : Static trace
  Environment    : Indoor lab with GNSS signal repeater
                   (Open-sky testing is not permitted at this facility.
                    A GNSS signal repeater is used as an alternative.)
  Location       : Bangalore, India (~13.067°N, 77.592°E)

  IMPORTANT NOTE ON TEST ENVIRONMENT:
  GNSS signal repeaters do not fully replicate open-sky signal
  characteristics. The repeater introduces correlated signal path delays
  and non-representative multipath conditions that are known to degrade
  pseudorange residual metrics. Failures in PR residual checks should
  therefore be interpreted in the context of this test environment
  limitation, and not solely as a device defect.

  FAILURE CLASSIFICATION
  ----------------------

  [*] Expected — Hardware Limitation (not a defect):

    - Required Signal Types (GPS L5 missing):
      The Sony XQ-GE54 does not support the GPS L5 frequency band.
      This failure is expected and is a known hardware capability
      boundary for this device.

    - Percent Valid ADR / Percent Usable ADR / ADR Residual Epochs:
      The device does not support ADR (Accumulated Delta Range /
      carrier phase) measurements. ADR-based checks are therefore
      not applicable. These failures are expected.

  [**] Under Investigation — Hardware Clock Gaps:

    - Delta ElapsedRealtimeNanos Range (1000.887 ms vs ≤1.0 ms)
    - Delta ElapsedRealtimeNanos Std   (161.52 ms vs ≤2.5 ms)

      These failures indicate large gaps between consecutive
      ElapsedRealtimeNanos timestamps. This pattern is typically
      associated with duty cycling being active on the device
      (the GNSS receiver periodically powers off between measurement
      epochs). The root cause on this specific unit is under
      investigation. No corrective action has been taken yet.

  [***] Test Environment Limitation — GNSS Signal Repeater:

    - L1-Freq PR Residuals Std          (100.00 m vs ≤6.5 m)
    - LS-Freq PR Residuals Std          (100.00 m vs ≤5.0 m)
    - L1-Freq Normalized PR Residual RMS (23.157 vs ≤5.0)

      Pseudorange residuals are significantly elevated. This is
      consistent with a repeater-based test environment:
        - The repeater re-broadcasts signals with additional fixed
          path delays not present in an open-sky scenario.
        - The positioning solver (WLS) produces large residuals when
          signal timing does not conform to the expected open-sky
          geometry model.
      These failures are attributed to the test environment and are
      not indicative of a device defect.

  SUMMARY — Device 1:
  All signal-strength, timing jitter, velocity (PRR), and measurement
  state checks pass. The device performs well within expected bounds
  for a device lacking L5 and ADR support, in a repeater environment.

  0 out of 32 conformance checks were completed.

  ROOT CAUSE: Duty Cycling Enabled on Device
  -------------------------------------------
  The Sony SO-51G had duty cycling (power-saving mode for GNSS) active
  during the recording session. Duty cycling causes the GNSS receiver
  hardware to periodically power off and restart to conserve battery.
  Each restart produces a large, abrupt discontinuity in the hardware
  clock (a "cycle slip"). When such slips occur too frequently, the
  reference notebook's pre-processing step explicitly rejects the log
  with the above error, as the discontinuities make it impossible to
  reliably compute pseudoranges or perform WLS positioning.

  The Google notebook enforces this check for all non-Wear-OS devices,
  as duty cycling indicates the device is not in a compliant GNSS
  measurement state for quality evaluation.

  Cascading Errors (post-abort):
  --------------------------------
  Following the fatal error at Cell 13, all subsequent cells failed
  with NameError / KeyError exceptions due to undefined variables
  (cc_results, df_wls, millis_to_ecef_m, etc.) — these are
  consequences of the early termination, not independent issues.

  ACTION REQUIRED
  ---------------
  The test for Sony SO-51G must be re-run with duty cycling disabled:

    1. On the device, go to:
       Settings → Apps → GnssLogger → Battery → Set to "Unrestricted"
    2. Keep the screen on during the entire capture session.
    3. Disable any adaptive battery or power-saving modes before
       starting the GnssLogger recording.
    4. Re-run the capture for a minimum of 120 seconds in the same
       repeater test environment.
    5. Re-execute gnss_analysis.ipynb with the new log.
