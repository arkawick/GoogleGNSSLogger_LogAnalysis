DESCRIPTION

BACKGROUND
----------
High-quality GNSS performance depends on three critical components: the GNSS
chip, oscillator, and antenna. Google provides a standardized toolset to
evaluate GNSS performance on Android devices:

  - GnssLogger App     : Collects raw GNSS measurements from the device.
  - gnss_analysis.ipynb: Google Colab notebook (v2026.01.29) that analyzes
                         collected logs against 32 conformance checks with
                         defined quality thresholds.

This ticket documents GNSS quality validation results for the Sony XQ-GE54
running Android 17, captured under two different network conditions (Airtel
and Jio SIM), in alignment with the Google Android Bootcamp GNSS Quality
Framework.


TEST SETUP
----------
  Tool (Logger)  : GnssLogger App (Android)
  Tool (Analysis): gnss_analysis.ipynb — Google Colab (v2026.01.29)
  Test type      : Static trace
  Environment    : Indoor lab with GNSS signal repeater
                   (Open-sky testing is not permitted at this facility.
                    A GNSS signal repeater is used as an alternative.)
  Location       : Bangalore, India (~13.067°N, 77.592°E)
  Capture date   : 2026-03-12

  IMPORTANT NOTE ON TEST ENVIRONMENT:
  GNSS signal repeaters do not fully replicate open-sky signal
  characteristics. The repeater introduces correlated signal path delays
  and non-representative multipath conditions that are known to degrade
  pseudorange residual metrics. Failures in PR residual checks should
  therefore be interpreted in the context of this test environment
  limitation, and not solely as a device defect.

    (carrier phase). All ADR-dependent checks return NaN or 0% and

    GPS L5 / GAL E5A missing: Device does not support L5 or E5A bands.
    All L5-dependent checks return NaN. Expected failures.

  [^] Borderline — BeiDou B1I Coverage:
    - BDS B1I signals/epoch: 0.4 (threshold ≥5.0)
    BeiDou B1I visibility is very low (0.4 avg per epoch). This may be
    a sky visibility or repeater signal coverage issue. BDS B1C tracks
    well (10.0/epoch), suggesting the chipset is functional but B1I
    satellites are not in a favourable position or are not rebroadcast
    by the repeater for this band.

  [**] Minor — ElapsedRealtimeNanos Range:
    - Delta ElapsedRealtimeNanos Range: 1.5 ms (threshold ≤1.0 ms)
    The range failure is marginal (1.5 ms vs 1.0 ms limit). The Std
    check passes (0.36 ms ≤ 2.5 ms), indicating this is likely a
    single outlier epoch rather than systemic duty cycling. No
    corrective action required at this stage.

  POSITIVE HIGHLIGHTS — Airtel:
    - L1 PR Residuals Std: 2.0 m (well within 6.5 m threshold)
      → Excellent pseudorange quality for a repeater environment.
    - PRR Residuals Std: 0.01 m/s — very low velocity noise.
    - Clock Bias Std: 1.275 m/s (within 2.0 m/s threshold).
    - All timing and signal count checks pass.

  [**] Test Environment + Hardware Clock — PR Residuals and Clock Gaps:
    - Delta ElapsedRealtimeNanos Range: 1001.050 ms (threshold ≤1.0 ms)
    - Delta ElapsedRealtimeNanos Std:   157.495 ms  (threshold ≤2.5 ms)
    - L1 PR Residuals Std:              115.87 m    (threshold ≤6.5 m)
    - LS-Freq PR Residuals Std:         115.87 m    (threshold ≤5.0 m)
    - Normalized PR Residual RMS:       29.416      (threshold ≤5.0)

    The clock gap magnitude on the Jio capture (~1001 ms range,
    ~157 ms std) is dramatically larger than the Airtel capture
    (1.5 ms range, 0.36 ms std), despite both being captured on the
    same device on the same day. This indicates an intermittent
    hardware clock discontinuity that appeared during the Jio session,
    consistent with duty cycling triggering during the recording.
    The large PR residuals (115 m) are a direct consequence of these
    clock gaps corrupting the pseudorange computation.

    Note: The PRR (velocity) checks pass on both captures, indicating
    the Doppler tracking remained stable even when the clock gaps
    occurred.

  KEY OBSERVATION:
  Both captures are from the same device on the same day. The Airtel capture
  shows good pseudorange quality (L1 PR residuals 2.0 m) while the Jio
  capture shows severely degraded pseudoranges (115.87 m). The root cause
  is the hardware clock gaps that occurred during the Jio session (duty
  cycling). This confirms the failures in the Jio log are not a SIM or
  network issue — they are caused by intermittent duty cycling behaviour
  on the device during that specific recording session.

        1. Settings → Apps → GnssLogger → Battery → Set to "Unrestricted"
        2. Keep screen on throughout the capture.
        3. Disable adaptive battery / power-saving before recording.
        4. Re-run for a minimum of 120 seconds and re-execute
           gnss_analysis.ipynb.

      pass at the same quality level as the Airtel capture.
